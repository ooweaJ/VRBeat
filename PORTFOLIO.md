## 1. 프로젝트 개요

VRBeat는 **VR 공간에서 전통 리듬게임의 "정박 타격감"을 구현**한 1인 개발 프로젝트입니다.

Beat Saber가 칼 방향과 시각적 박진감에 집중하는 것과 달리,  
VRBeat는 **소리와 타이밍이 칼각으로 딱 맞아떨어지는 순간의 쾌감**을 핵심 재미로 삼습니다.  
노트를 "어떻게 베느냐"보다 **"언제 베느냐"** — 박자의 정확성이 플레이어에게 직접 전달되도록 설계했습니다.

```
Beat Saber   →  방향·궤적 중심, 칼질 자체의 타격감
VRBeat       →  박자·타이밍 중심, 리듬에 정확히 맞춘 타격감
```

## 2. 차별화 포인트

| | Beat Saber | VRBeat |
|---|---|---|
| 타격 방식 | 지정된 방향으로 베기 | 방향 없이 박자에 맞춰 베기 |
| 핵심 재미 | 칼질·움직임의 박진감 | 리듬 정확도, 정박 쾌감 |
| 노트 접근 | 양쪽에서 색상별로 | 전통 리듬게임처럼 선에 도달 |
| 싱크 보정 | 고정 오프셋 | 사용자 맞춤 캘리브레이션 |
| 채보 제작 | 수작업 / 유료 툴 | BeatSage + 자체 변환기 |

## 3. 핵심 기술 도전 — 오디오 싱크 문제

### 3-1. 문제 정의

VR 리듬게임의 가장 큰 기술적 난관은 **오디오 출력 지연(Audio Output Latency)** 입니다.

Meta Quest 3의 오디오 레이턴시는 **120~160ms** 수준으로,  
개발자가 별도로 보정하지 않으면 음악은 노트보다 늦게 들립니다.  
타이밍 게임에서 이 오차는 즉시 "느낌이 안 맞는다"는 감각으로 전달됩니다.

```
[노트가 선에 닿는 시각] ──── 120ms ────▶ [플레이어가 소리를 듣는 시각]
                                          ↑
                           이 간격이 "싱크가 안 맞는다"는 느낌의 원인
```

### 3-2. 기술적 접근 — DSP 기반 타이밍

Unity의 `AudioSettings.dspTime`은 오디오 엔진의 절대 시간으로,  
프레임레이트 변동과 무관하게 **샘플 단위 정밀도**를 가집니다.

```csharp
// 노트 위치 계산 — noteSpeed는 시각적 거리만 결정, 타이밍에는 영향 없음
double songBeat = (dspTime - dspStartTime - songOffset - userOffset) * bpm / 60.0;
float zPos = hitZ + (float)((note.beat - songBeat) * secondsPerBeat * noteSpeed);

// 클릭 사운드 정밀 스케줄 — 프레임 드랍에도 정확한 타이밍 유지
audioSource.PlayScheduled(hitDspTime);
```

핵심은 **노트 속도(noteSpeed)가 타이밍과 완전히 분리**된다는 것입니다.  
속도를 바꿔도 히트 판정 시간은 달라지지 않습니다.

### 3-3. 캘리브레이션 시스템 — 3번의 설계 변경

#### ① 1차 시도: 단순 탭 측정

메트로놈 소리에 맞춰 8번 탭 → 중앙값을 오프셋으로 저장

**문제 발견:** 리듬에 맞춰 누를 때 사람은 자연스럽게 **예측해서 미리** 누릅니다.  
측정값이 실제 레이턴시보다 낮게 나오는 **예측 편향(Anticipation Bias)** 발생.  
70ms로 측정돼도 실제로는 여전히 빠르게 느껴지는 이유였습니다.

#### ② 2차 시도: 비주얼 트랙 + 탭

노트가 선에 닿는 시각을 보고 타이밍을 맞추는 방식

**문제 발견:** 이번엔 반대로 시각 단서를 보고 예측해서 **음수(-) 오프셋**이 나올 수 있음.  
시각 정보가 오히려 순수한 오디오 레이턴시 측정을 방해합니다.

#### ③ 최종 설계: 2단계 분리

```
┌─────────────────────────────────────────────────────┐
│  1단계 — 소리만 (시각 없음)                           │
│  딸깍 소리가 들린 후 누르기 × 5                       │
│  → 순수 오디오 레이턴시만 측정                        │
│  → "예측해서 미리 누르지 마세요" 안내 명시            │
├─────────────────────────────────────────────────────┤
│  2단계 — 비주얼 트랙 무한 루프                        │
│  측정된 오프셋 적용 후 노트가 무한 반복               │
│  → 소리와 시각이 맞는지 사용자가 직접 눈으로 확인    │
│  → [맞아요! 저장] 또는 [다시 측정] 선택              │
└─────────────────────────────────────────────────────┘
```

측정(1단계)과 검증(2단계)을 분리함으로써  
예측 편향을 제거하고 사용자가 직접 결과를 확인할 수 있도록 했습니다.

## 4. BeatSage 채보 변환 파이프라인

### 4-1. 문제

Beat Saber 맵(BeatSage 자동 생성)은 ML 기반 온셋 감지로 박자를 추출하기 때문에  
`2.7186`, `5.4372` 같은 **소수점 박자값**이 나옵니다.  
이를 그대로 쓰면 박자가 "착착" 맞는 느낌이 없고 리듬감이 흐릿해집니다.

### 4-2. 해결 — Beat Quantization

```python
def quantize_beat(beat, grid=0.25):
    """박자를 1/4 단위로 스냅"""
    return round(round(beat / grid) * grid, 4)

# 2.7186 → 2.75  (가장 가까운 1/4박자로 스냅)
# 5.4372 → 5.50
```

0.25(1/4박자) 그리드로 양자화하면 실제 음악의 박자에 정확히 맞아떨어집니다.  
테스트 결과 원본보다 **리듬감이 명확하게 개선**되는 것을 확인했습니다.

### 4-3. 변환 파이프라인 전체 흐름

```
BeatSage.com 다운로드
      ↓
beatsaber_to_vrbeat.py
  · Info.dat  → BPM, songOffset, title, artist 추출
  · *Standard.dat → 노트 배열 파싱 (v2/v3 양쪽 지원)
  · 박자 퀀타이즈 (0.25 그리드)
  · 롱노트 자동 삽입 (HPSS 하모닉 분석, librosa)
  · 오디오 파일 복사
      ↓
StreamingAssets/Songs/{song_id}/
  ├─ info.json
  ├─ chart_easy.json
  ├─ chart_normal.json
  └─ chart_hard.json
      ↓
Unity에서 Ctrl+R → SongSelect에 즉시 반영
```

## 5. 구현된 주요 시스템

### 게임플레이
- **Conductor** — DSP 기반 정밀 박자 계산, 프레임 독립적
- **NoteSpawner** — 오브젝트 풀링(64개), 비트 스케줄링
- **HealthSystem** — HP 바, 롱노트 유지 판정, 자르면 회복
- **ScoreSystem** — Perfect/Good/Miss 판정, 콤보 배율

### UI/UX
- **SongSelectUI** — 곡 목록, 난이도 선택, 최고점 표시
- **ResultUI** — 점수, 랭크, 통계 표시
- **CalibrationController** — 2단계 맞춤 오프셋 측정
- **SettingsUI** — 노트 속도, 볼륨, 오프셋 수동 조정

### VR 특화
- **World-space Canvas** — HMD 앞 2.5m 고정 UI
- **XR Interaction Toolkit** — 레이캐스터 기반 UI 인터랙션
- **SkyboxVideo** — 배경 영상 스카이박스

## 6. 개발 과정에서 배운 것

### 오디오 타이밍의 복잡성
게임 로직과 오디오 엔진의 시간 도메인은 다릅니다.  
`Time.time`은 프레임마다 달라지지만 `AudioSettings.dspTime`은 샘플 단위로 진행됩니다.  
리듬게임에서는 반드시 DSP 시간 기반으로 판정을 구현해야 합니다.

### 사용자 인식과 실제 레이턴시의 차이
하드웨어 레이턴시를 측정하는 것과 **사용자가 느끼는 싱크**는 다릅니다.  
인간의 예측 심리, 시각/청각 선행 관계가 모두 "맞다/틀리다" 판단에 영향을 줍니다.  
단순 수치 측정만으로는 부족하고, **사용자가 직접 확인하는 검증 단계**가 필수입니다.

### 채보 품질이 게임 재미를 결정한다
노트 타이밍이 실제 음악의 박자와 조금만 어긋나도 리듬감이 완전히 사라집니다.  
자동 생성 채보를 그대로 쓰지 않고 **후처리(퀀타이즈)** 를 추가한 이유입니다.
